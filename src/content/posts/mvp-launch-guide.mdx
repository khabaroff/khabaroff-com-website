# Запуск MVP за неделю: практический гайд

MVP — это не плохая версия продукта. Это самый быстрый способ проверить гипотезу.

<Callout type="info">
**MVP (Minimum Viable Product)** — минимальный продукт, который решает одну главную проблему пользователя.
</Callout>

## Почему неделя?

**Дольше = опаснее:**
- Тратите деньги впустую
- Теряете мотивацию
- Рынок меняется
- Конкуренты запускаются быстрее

**Неделя заставляет фокусироваться.**

## День 1: Определение проблемы

### Не начинайте с решения

**Плохо:**
"Хочу сделать мобильное приложение для..."

**Хорошо:**
"Люди тратят 3 часа в неделю на [задачу], потому что [текущее решение] неудобно"

### Вопросы для проверки

- Проблема болит прямо сейчас?
- Люди уже платят за решение?
- Вы сами столкнулись с этой проблемой?
- Можете найти 5 человек с такой проблемой за день?

<Callout type="warning">
**Если на хотя бы 2 вопроса ответ "нет"** — проблема недостаточно серьезная.
</Callout>

### Практика

**Задание на день:**
1. Опишите проблему в 1 предложении
2. Найдите 5 человек с этой проблемой
3. Поговорите с ними (звонок, не текст!)
4. Запишите ключевые фразы, которые они используют

## День 2: Определение решения

### Одна фича

MVP решает ОДНУ проблему ОДНИМ способом.

**Пример плохого MVP:**
"Платформа для фрилансеров с портфолио, чатом, оплатой, отзывами и календарем"

**Пример хорошего MVP:**
"Автоматическое выставление счетов для фрилансеров"

### Smoke test (тест дымом)

Проверьте спрос ДО разработки:

1. **Landing page** — описание продукта + кнопка "Купить"
2. **Waitlist** — "Скоро запустимся, оставьте email"
3. **Pre-sale** — "Купите сейчас со скидкой, получите через неделю"

<Callout type="success">
**Цель:** Если 0 людей не заинтересовались лендингом — 0 людей будут пользоваться продуктом.
</Callout>

### Практика

**Задание на день:**
1. Опишите решение в 1 предложении
2. Создайте лендинг (Tilda, Webflow, Carrd)
3. Запустите рекламу на $50
4. Цель: 10 email в waitlist

## День 3: Выбор стека

### No-code > Low-code > Code

**No-code варианты:**

**База данных + UI:**
- Airtable + Softr
- Notion + Super
- Google Sheets + Glide

**Автоматизация:**
- Zapier / Make
- n8n (open-source)

**Формы + оплата:**
- Tally + Stripe
- Typeform + PayPal

**All-in-one:**
- Bubble (полноценные web apps)
- Adalo (мобильные приложения)

### Когда нужен код

Только если:
- No-code физически не может сделать функцию
- Критична скорость работы
- Нужна кастомная логика

**Быстрый стек для кода:**
- Frontend: Astro / Next.js
- Backend: Supabase / Firebase
- Auth: Clerk / Supabase Auth
- Payments: Stripe
- Deploy: Vercel / Netlify

### Практика

**Задание на день:**
1. Выберите no-code стек
2. Создайте прототип главного экрана
3. Проверьте что можно кликнуть

## День 4-5: Разработка core фичи

### Фокус на happy path

**Happy path** — идеальный сценарий использования.

**НЕ делайте:**
- Обработку всех ошибок
- Восстановление пароля
- Уведомления на email
- Темную тему
- Мобильную версию (если основная аудитория на desktop)

**Делайте:**
- Один главный сценарий от начала до конца
- Базовый UI (можно некрасиво)
- Минимальную валидацию
- Ручные процессы вместо автоматизации

<Callout type="info">
**Правило:** Если функцию можно сделать вручную — делайте вручную.
</Callout>

**Примеры:**

| Автоматизация | Ручной способ |
|---------------|---------------|
| Email уведомления | Отправляете вручную |
| Автоматический экспорт | Скачиваете и отправляете сами |
| AI рекомендации | Даете рекомендации сами |

### Практика

**Задание на 2 дня:**
1. Реализуйте 1 главный сценарий
2. Протестируйте на себе
3. Дайте протестировать другу
4. Исправьте критичные баги

## День 6: Подготовка к запуску

### Checklist перед запуском

**Технический:**
- ☐ Работает happy path от начала до конца
- ☐ Есть способ связаться (email, Telegram)
- ☐ Настроена аналитика (хотя бы Google Analytics)
- ☐ Протестировано на разных браузерах

**Бизнес:**
- ☐ Понятно как клиент платит
- ☐ Ясна ценность продукта
- ☐ Есть plan B если сервер упадет

**Коммуникация:**
- ☐ Landing page объясняет проблему и решение
- ☐ Есть призыв к действию (CTA)
- ☐ Работает форма обратной связи

### Ценообразование

**Для MVP можно:**

1. **Free tier** — пусть попробуют бесплатно
2. **Фиксированная цена** — $10-50/месяц
3. **Pay what you want** — платите сколько считаете справедливым

<Callout type="warning">
**Не делайте:** Сложные тарифные планы. Один тариф достаточно.
</Callout>

### Практика

**Задание на день:**
1. Пройдите чеклист
2. Попросите 3 друзей протестировать
3. Подготовьте текст для анонса

## День 7: Запуск

### Где анонсировать

**Обязательно:**
- Ваши соцсети
- Email тем, кто в waitlist
- Комьюнити где тусуется ваша аудитория

**Опционально:**
- Product Hunt
- Hacker News (Show HN)
- Reddit (подходящие сабреддиты)
- Twitter с #buildinpublic

### Формат анонса

**Структура:**

1. **Проблема** — что не так сейчас
2. **Решение** — что вы сделали
3. **Призыв** — попробуйте и дайте фидбек
4. **Ссылка** — куда идти

**Пример:**

```
Запустили [название] — [решение] для [аудитории]

Проблема: [проблема]

Решение: [как работает в 1 предложении]

Сейчас это MVP, но уже работает!

Буду благодарен за фидбек: [ссылка]
```

### Что делать после запуска

**Первые 24 часа:**
- Отвечайте на ВСЕ комментарии
- Собирайте фидбек
- Фиксите критичные баги

**Первая неделя:**
- Звоните каждому early adopter
- Спрашивайте что не так
- Итерируйте

<Callout type="success">
**Цель MVP:** Не продать, а научиться. 10 разговоров с пользователями важнее 100 signup.
</Callout>

## Частые ошибки

**Перфекционизм**
"Запущу когда будет идеально" = никогда не запущу

**Слишком много фич**
MVP с 20 фичами = не MVP

**Игнорирование фидбека**
Пользователи говорят что не так — слушайте!

**Сдаться после первой недели**
MVP — это начало, а не конец.

## Заключение

Неделя — это реально.

**Секрет:** Фокус на одной проблеме и одном решении.

**План действий:**
1. День 1: Проблема
2. День 2: Решение + лендинг
3. День 3: Стек
4. День 4-5: Разработка
5. День 6: Подготовка
6. День 7: Запуск

Готов поспорить — вы откладываете запуск из страха, а не из-за сложности.

**Действуйте.**
