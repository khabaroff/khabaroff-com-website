# Создание дизайн-системы с нуля

Дизайн-система — это не просто набор компонентов. Это язык, на котором говорит ваш продукт.

## Зачем нужна дизайн-система

<Callout type="info">
  **Дизайн-система** — это набор стандартов, компонентов и правил, которые обеспечивают консистентность продукта.
</Callout>

**Преимущества:**

* Скорость разработки ↑
* Консистентность ↑
* Коммуникация команды ↑
* Technical debt ↓
* Время onboarding новичков ↓

## Когда создавать

**НЕ создавайте, если:**

* Продукт на стадии прототипа
* Команда \< 5 человек
* Меньше 10 экранов
* Нет времени на поддержку

**Создавайте, если:**

* Стабильный продукт
* Команда дизайнеров + разработчиков
* 20+ экранов
* Планы на масштабирование

## Этапы создания

### 1. Аудит текущего UI

Соберите все экраны и выпишите:

* Используемые цвета
* Размеры шрифтов
* Отступы
* Компоненты (кнопки, инпуты, карточки)
* Паттерны (формы, списки, модалы)

**Инструмент:** Figma → плагин "Design Lint" для аудита

### 2. Определите токены

**Токены** — атомарные значения дизайна.

```css
/* Цвета */
--color-primary: #ed6eac;
--color-bg: #fff6f8;
--color-text: #222222;

/* Типографика */
--font-size-base: 16px;
--font-size-lg: 20px;
--font-size-xl: 24px;

/* Spacing */
--space-xs: 4px;
--space-sm: 8px;
--space-md: 16px;
--space-lg: 24px;
```

<Callout type="success">
  **Совет:** Используйте семантические имена (primary, success) вместо буквальных (pink, green).
</Callout>

### 3. Создайте базовые компоненты

**Atomic Design иерархия:**

1. **Atoms** (атомы): кнопка, инпут, иконка
2. **Molecules** (молекулы): поле формы с лейблом
3. **Organisms** (организмы): форма логина
4. **Templates** (шаблоны): layout страницы
5. **Pages** (страницы): конкретная страница

Начните с атомов.

### 4. Документируйте компоненты

Для каждого компонента опишите:

* **Варианты** (states): default, hover, disabled
* **Размеры**: small, medium, large
* **Props**: что можно настроить
* **Когда использовать**: use cases
* **Accessibility**: ARIA атрибуты

<Callout type="warning">
  **Важно:** Недокументированная система = мертвая система. Документация должна жить вместе с кодом.
</Callout>

## Инструменты

### Дизайн

**Figma:**

* Variables для токенов
* Auto Layout для responsive
* Components с вариантами
* Shared Libraries

**Альтернативы:** Sketch, Penpot (open-source)

### Разработка

**Storybook:**

* Изолированная разработка компонентов
* Визуальная документация
* Тестирование UI
* Интеграция с Figma

**Код:**

```bash
# Установка Storybook
npx storybook@latest init

# Создание компонента
npm run storybook
```

### Документация

**Zeroheight** — связывает Figma с документацией
**Notion** — для процессов и guidelines
**MDX** — документация в коде

```

    └── guidelines.md
```

## Принципы хорошей системы

**Гибкость:**

* Компоненты можно комбинировать
* Props для кастомизации
* Escape hatches для исключений

**Ограничения:**

* Четкие варианты использования
* Запрет на прямые стили
* Валидация props

## Governance

**Кто управляет системой?**

Опции:

1. **Федеративная** — каждая команда контрибьютит
2. **Централизованная** — выделенная команда
3. **Гибридная** — core team + контрибьютеры

<Callout type="info">
  **Рекомендация:** Начните с централизованной модели, затем откройте для контрибуций.
</Callout>

## Процесс добавления компонента

1. **Предложение:** GitHub issue с use case
2. **Обсуждение:** Команда оценивает необходимость
3. **Дизайн:** Создание в Figma
4. **Review:** Feedback от команды
5. **Разработка:** Реализация в коде
6. **Документация:** Storybook + гайдлайны
7. **Release:** Versioning + changelog

## Версионирование

Используйте SemVer:

* **Major** (1.0.0 → 2.0.0): breaking changes
* **Minor** (1.0.0 → 1.1.0): новые фичи
* **Patch** (1.0.0 → 1.0.1): bug fixes

## Метрики успеха

Как понять, что система работает:

* % переиспользуемых компонентов
* Время создания нового экрана
* Количество UI багов
* Скорость onboarding дизайнеров

<Callout type="success">
  **Совет:** Не копируйте чужие системы. Создавайте под свои нужды, вдохновляйтесь принципами.

## Частые ошибки

* Создание всех компонентов сразу (начните с 5-7)
* Слишком гибкие компоненты (делайте opinionated)
* Нет владельца системы (должна быть ответственность)
* Игнорирование accessibility
* Отсутствие миграционных путей   

## Заключение

Дизайн-система — это инвестиция. Первые месяцы будет медленнее, потом окупится скоростью.

**План действий:**

1. Недели 1-2: Аудит + токены
2. Недели 3-4: Базовые компоненты (5-7 штук)
3. Недели 5-6: Документация + Storybook
4. Неделя 7+: Внедрение в продукт

Начните маленько, растите постепенно. Главное — консистентность и поддержка.
